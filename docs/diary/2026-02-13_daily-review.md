## 잘한 것

### 1. 계획 준수 — 4일 만에 처음
어제까지 3일 연속 "계획 세우고 안 지킴" 피드백 받았는데, 오늘은 1→2→3→4단계 순서 그대로 실행. 중간에 딴 길로 새지 않았음.

### 2. 사용자 가치에 집중
- 공개 청첩장 뷰 완성 = **실제 서비스 사용 가능 상태** 도달
- 결제 설계 완료 = **수익화 경로 확보** (MVP에 필요한 마지막 퍼즐)
- 더 이상 "사용자 가치 제로"가 아님

### 3. 기술 부채 동시 처리
핫픽스 4건 + 코드 품질 3건 = 7개 이슈를 새 기능 추가 없이 정리. 특히:
- 크레딧 감사 추적: 운영 투명성 확보
- SSE + 이미지 URL 검증: 보안 취약점 사전 차단
- CASCADE→RESTRICT: 법적 데이터 보관 요건 충족

### 4. 결제 설계 방향 전환 판단
Toss → PortOne V2 + 네이버페이 변경은 단순 PG 교체가 아니라:
- PG aggregator 도입으로 향후 결제수단 추가 용이
- 스마트스토어 채널 추가로 네이버 유입 경로 확보
- 설계 문서 + beads 이슈 9개 체계적 등록

## 아쉬운 것

### 1. 코드 없는 설계 = 실제 검증 안 됨
결제 설계가 문서로는 깔끔하지만, 실제 PortOne SDK 붙여보기 전까지는 확신 못 함. 특히:
- PortOne V2 + 네이버페이 bypass 파라미터가 문서대로 동작하는지
- Commerce API bcrypt 인증이 Node.js에서 매끄럽게 되는지
- 네이버페이 검수 기간 (15영업일)이 일정에 미치는 영향

### 2. 스마트스토어 활성화 UX 한계
"주문번호 직접 입력" 방식은 기술적으로 확실하지만 UX가 좋지 않음. 구매자가:
1. 스마트스토어에서 주문번호 복사
2. 쿠구 사이트 이동
3. 로그인
4. 주문번호 붙여넣기

4단계나 되는 과정. 나중에 스마트스토어 트래픽이 생기면 개선 필요.

### 3. 아직 결제 코드 한 줄도 없음
설계만 4단계에서 했고 구현은 0%. Phase 1만 해도 6개 이슈. 이게 진짜 돌아가려면 최소 1~2일 더 필요.
